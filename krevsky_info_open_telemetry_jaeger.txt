Общая схема:
1) service provides traces to otel-collector
2) otel-collector aggregates, batches, does sampling, retries during sending to jaeger

Как подключить jaeger и otel-collector:
1) скачать джарник https://github.com/open-telemetry/opentelemetry-java-instrumentation/releases/tag/v2.27.0
2) создать контейнеры jaeger и otel-collector в docker-compose
3) создать otel-collector-config.yaml
4) для каждого сервиса в application.properties прописать:
    ### to use OpenTelemetry and jaeger to see traces in distributed system
    # turn on Spring Boot 3 Micrometer Tracing (NOTE: usual value for production 0.1 .. 0.01)
    management.tracing.sampling.probability=1
    management.otlp.tracing.endpoint=http://otel-collector:4318/v1/traces
    # to save trace_id and span_id
    logging.pattern.level: "%5p [${spring.application.name:},traceId=%X{trace_id:-},spanId=%X{span_id:-}]"
5) для контейнеров каждого сервиса добавить:
    копирование джарника opentelemetry-javaagent.jar в контейнер
    volumes:
          - ./monitoring/opentelemetry-javaagent.jar:/otel/opentelemetry-javaagent.jar
    java опции, чтобы пушить данные для трейсинга в коллектор
      JAVA_TOOL_OPTIONS: >
        -javaagent:/otel/opentelemetry-javaagent.jar
        -Dotel.service.name=inventory-service
        -Dotel.exporter.otlp.endpoint=http://otel-collector:4318
        -Dotel.exporter.otlp.protocol=http/protobuf
    зависимость от контейнера otel-collector:
    depends on:
      - otel-collector

OTLP бывает двух типов
1. gRPC - port 4317
2. HTTP protobuf (http/protobuf) port 4318
а порт 8889 - prometheus scrapes otel-collector:8889/metrics (т.е. метрики самого коллектора)

Collector — это observability gateway

-------------------------------------
telemetry |	transport model
-------------------------------------
traces	  |	push
metrics	  |	usually pull (prometheus)
logs	  |	push
-------------------------------------

строка
    -Dotel.exporter.otlp.protocol=http/protobuf
говорит, какой transport protocol использовать

jaeger UI: http://localhost:16686

best practice:
    1) выносить все observability контейнеры в отдельный docker-compose-observability.yaml файл
        и запускать
        docker compose -f docker-compose.yaml -f docker-compose-observability.yaml build
        docker compose -f docker-compose.yaml -f docker-compose-observability.yaml up -d
        Эта команда сначала мержит файлы в итоговую модель, а потом уже запускает
    2) NOTE: по умолчанию НЕТ макс ограничения трейсов, хранищихся в RAM jaeger-a,
        поэтому он будет потреблять неограниченное кол-во памяти!
        НАДО
            environment:
              # Жёстко ограничиваем количество трассировок в памяти
              - MEMORY_MAX_TRACES=5000  # вместо бесконечности
              deploy:
                resources:
                  limits:
                    memory: 512m
                  reservations:
                    memory: 256m  # "мягкий" лимит — приоритет при нехватке

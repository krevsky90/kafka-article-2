Идемпотентность для продюсера:
# ADD PRODUCER IDEMPOTENCY
# 0   - at-most once
# 1   - at-least once
# all - at-least once, all is MANDATORY for idempotency!
spring.kafka.producer.acks=all
# based on ProducerId + ProducerEpoch (is incremented when producer is re-created) + sequenceNumber (number of message sent by this producer)
# if true => sets transactional.id - broker assigns unique ProducerId + ProducerEpoch to the producer
# example from logs:
# 2026-04-16T14:40:25.884Z " INFO [inventory-service]" 1 --- [inventory-service] [vice-producer-1] o.a.k.c.p.internals.TransactionManager   : [Producer clientId=inventory-service-producer-1] ProducerId set to 12000 with epoch 0
spring.kafka.producer.properties.enable.idempotence=true
# how many times producer sends message without getting ack. has to be > 0
spring.kafka.producer.retries=2147483647
# max amount of messages that producer can send before getting ack regarding some of these messages. needs to be <= 5
spring.kafka.producer.properties.max.in.flight.requests.per.connection=5

    - бизнес логика

2) в сервисе консюмера добавить бин с @Scheduled-методом, к-ый берет не отправленные эвенты из outbox_events и пытается их отправить в роли продюсера (at-least once)
    отправка д б СИНХРОННОЙ! иначе если сообщение в кафка не пришло, но шедулер-бин пометил сообщение в outbox_events как отправленное, то оно не будет переотправляться => потеряется
+ добавить @EnableScheduling в главный класс сервиса

=== ОФФСЕТЫ ===
info: https://habr.com/ru/articles/965218/
типы коммитов:
1) авто-коммит - по умолчанию
2) ручной:
    BATCH
    RECORD
    MANUAL
    MANUAL_IMMEDIATE
для этого надо проставить spring.kafka.listener.ack-mode в application.properties консюмера.
+ надо отключить spring.kafka.consumer.enable-auto-commit

По типу синхронности:
1) Синхронный
2) Асинхронный

NOTE: Spring Boot по умолчанию использует для автокоммита асинхронный режим, а для ручных коммитов — синхронный.

ПРОБЛЕМА: в InventoryService происходит коммит в БД и коммит оффсета.
    Чтобы не было ситуации "оффсет закоммичен, но коммит в БД упал", НЕ НАДО вешать обе аннотации @KafkaListener и @Transactional на один метод.
РЕШЕНИЕ:
    НАДО, чтобы коммит оффсета был строго ПОСЛЕ коммита в БД.
    Для этого создаем отдельный метод с @Transactional в ОТДЕЛЬНОМ (!) бине (из-за особенностей спринга)
    и вызываем его в методе с @KafkaListener.

    ИЛИ все в одном классе-листенере
        import org.springframework.transaction.support.TransactionTemplate;

        private final TransactionTemplate transactionTemplate;

        @KafkaListener(topics = "order-placed")
        public void reserveInventory(OrderPlacedEvent orderPlacedEvent) {
            transactionTemplate.executeWithoutResult(status -> {
                processOrderInTransaction(orderPlacedEvent);
            });
        }

        private void processOrderInTransaction(OrderPlacedEvent orderPlacedEvent) {
            ...
        }

=== ОБРАБОТКА ОШИБОК И Dead Letter Queues ===
info: https://habr.com/ru/articles/989608/

1) создать -dlt топики для существующих топиков:
    там, где создаем обычный топик, там же (в том же KafkaConfig) создаем и соотв-ющий dlt-топик
2) в сервисе-консюмере (inventory-service) заменим value-deserializer на ErrorHandlingDeserializer
    + укажем delegate десериалайзер
3) в сервисе-консюмере (inventory-service) создаем бин public DefaultErrorHandler errorHandler,
    где указываем, в какой топик идет проблемное сообщение
4) создаем dlt-processor-service, слушающий все dlt-топики и что-то делающий с их сообщениями

Как проверить:
    отправить кривое сообщение (через kafka-ui) в order-placed и убедиться, что
    - это сообщение в order-placed и в order-placed-dlt
    - это сообщение залоггировано в dlt-processor-service
    - этого сообщения нет в processed_order_id таблице inventory_db

Добавить Retryable и NonRetryable исключения - свои классы:
    RetryableException → инфраструктурные проблемы (DB, сеть)
    NonRetryableException → данные/логика (сериализация, invalid payload)
    тогда
    если вылетело RetryableException, то будут ретраи согласно настройкам в FixedBackOff
        если не поможет - сообщение отправляется в DLT
    если вылетело NonRetryableException, то сообщение СРАЗУ отправляется в DLT

NOTE: по умолчанию все исключения кафка считает retryable

=== Apache Avro и Schema Registry ===
info: https://habr.com/ru/articles/1018944/

Schema Registry хранит схемы не внутри себя, а в специальном топике _schemas. То есть кластер Kafka используется как база данных

Пример value из сообщения из order-service:
JSON:
    {
    	"orderId": "3f05673f-1b61-4a7a-b7ae-bd649e4ea523",
    	"email": "krevsky@gmail.com",
    	"productName": "Tablet",
    	"quantity": 1
    }
Header:
    {
    	"__TypeId__": "io.mitochondria.events.order.OrderPlacedEvent"
    }

Avro:
    H1d8f1260-3496-4460-88d2-d229ac6923b0"krevsky@gmail.comTablet
Header:
    пустой

NOTE: если для kafka-ui настроено KAFKA_CLUSTERS_0_SCHEMAREGISTRY: http://schema-registry:8081,
    то в kafka-ui сообщение показывается в виде JSON-а (для удобства), т.к.
        Kafka UI делает:
        читает сырые байты из Kafka
        идёт в Schema Registry
        получает схему
        десериализует Avro
        показывает тебе как JSON
    на самом деле сообщение в авро-виде, в чем можно убедиться через консоль:
        docker exec -it kafka-0 kafka-console-consumer --bootstrap-server kafka-0:9091 --topic order-placed --from-beginning

NOTE: внимательно следить за тем, что именно отправляется в кафку! это д б Avro-формат, а не JSON-овые DTO!
    иначе будут ошибки вида: org.apache.kafka.common.errors.SerializationException: Error registering Avro schema"bytes"
    тогда внимательно проверяем в коде, что отправляем в kafkaTemplate.send

NOTE: внимательно следить за тем, чтоб в БД сохранять именно Dto (на базе json-а), а не авро

NOTE: т.к. Dead letter processor принимает данные от ErrorHandlingDeserializer, к-ый отправляет baase-64 строки,
    то deserializer в Dead letter processor-е должен быть StringDeserializer, без всякого авро.
    т.е. best practice: сервисы общаются авро-сообщениями, а в DL-processor обрабатывает строки

Мавен команды:
1) чтобы собрать отдельно модуль мультимодульного приложения:
    mvn clean package -pl inventory-service,someOtherModule (-am)
        где -am (also make) — подтянет и соберёт зависимости этого модуля (если service1 зависит от других модулей внутри проекта)

Докер команды:
1) Если изменил код сервиса inventory-service:
    # 1. Собрал jar
    mvn clean package -DskipTests
    # 2. Пересобрал сервис
    docker-compose build inventory-service
    # 3. Перезапустил
    docker-compose up -d --no-deps inventory-service
    или (если несколько контейнеров надо)
    docker-compose up -d --no-deps --scale inventory-service=2
    иои (100% создание из новых имаджей)
    docker-compose up -d --no-deps --build --force-recreate --scale inventory-service=2
    где --no-deps значит, что не трогаются остальные сервисы нашего docker-compose
2) Если запускаю несколько инстансов сервиса, то, например,
    docker-compose up --scale inventory-service=2 --scale notification-service=2

Кафка:
1) как посмотреть оффсеты для партиции и консюмер группы:
    зайти на кафку: docker exec -it kafka-0 bash
    выполнить: kafka-consumer-groups --bootstrap-server kafka-0:9091 --group inventoryService --describe
    ИЛИ
    в одну команду извне:
    docker exec -it kafka-0 kafka-consumer-groups --bootstrap-server kafka-0:9091 --group inventoryService --describe
2) как подвинуть оффсет для консюмера:
    а) остановить консюмер
        иначе при попытке изменить оффсет будет ошибка:
            Error: Assignments can only be reset if the group 'notificationService' is inactive, but the current state is Stable
    б) узнать offset сообщения, начиная с которого нужно проиграть еще раз:
        можно посмотреть в kafka-ui
        или посмотреть offset lag (если проблема в ранее сломанном консюмере) консюмера для конкретного topic:partitionNumber и вычесть его из оффсета партиции:
            docker exec -it kafka-0 kafka-consumer-groups --bootstrap-server kafka-0:9091 --group notificationService --describe
    в) прогнать команду по изменению оффсета БЕЗ --execute, чтобы просто увидеть, что будет выполнено.
        а затем прогнать команду с --execute:
            docker exec -it kafka-0 kafka-consumer-groups --bootstrap-server kafka-0:9091 --group notificationService --topic inventory-reserved:2 --reset-offsets --to-offset 5 --execute
                где inventory-reserved:2 значит топик inventory-reserved, партиция 2

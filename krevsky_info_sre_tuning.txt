КАКАЯ взаимосвязь между параметрами сервиса:

ЕСЛИ мало памяти, то
1) всем потокам не хватает памяти (см выше про server.tomcat.threads.max)
2) при запросах к сервису тратится RAM на создание (request/response objects, DTO, Entity, JSON serialization, buffers, thread stacks, caches, Hibernate persistence context, Netty/Tomcat buffers)
    чем больше RPS, тем больше быстрее заполняется Head,
    тем чаще и агрессивнее вызывается GC (будут бОльшие GC pauses (и Stop the world) + CPU начинает сильнее расходоваться на GC)
    тем больше простаивают Tomcat потоки, ожидая окончания работы GC,
    тем сильнее растет (время обработки запросов),
    тем больше потоков будет создавать Tomcat (up to server.tomcat.threads.max), чтобы обработать входящие запросы,
    а после этого запросы идут в accept queue, а потом possible reject/timeouts
ИТОГО: Если GC не успевает освобождать память:
- heap переполняется
- начинается allocation failure
- возможен OOMKilled / OutOfMemoryError
- сервис падает.

ЕСЛИ мало CPU (упирается/касается 1.0), то
1) т.к. CPU расходуется на работу потоков Томката (business logic, serialization, DB drivers, GC, encryption, logging, Kafka clients, context switching),
    будет вызван CPU throttling (т.е. ядро linux насильно усыпляет процессы сервиса),
    процессы получают меньше CPU time,
    из-за этого резко вырастет Duration/Latency (время обработки запросов)
2) при высокой загрузке CPU:
    GC может не успевать
    request processing замедляется
    растут очереди и время ожидания
ИТОГО: НЕ должно быть упирания в лимит 1.0 от выделенного!

ЕСЛИ много потоков, то
1) Растет:
    context switching
    scheduler overhead
    lock contention
    потребление stack memory
2) Потоки начинают конкурировать за:
    CPU
    locks
    memory bandwidth
3) Из-за большого количества одновременно работающих потоков:
    увеличивается allocation rate
    растет нагрузка на GC
    Это может приводить к:
    росту latency
    снижению throughput
    ухудшению tail latency (p95/p99)
4) Большое количество потоков не всегда увеличивает производительность.
    После определенного предела начинается деградация.

Если слишком мало потоков
1) При:
    медленных запросах
    slowdown БД
    внешних API
    всплесках нагрузки
worker threads могут быстро исчерпаться (это называется SATURATION !!!)
2) Новые запросы начинают:
    ждать в очереди
    получать timeout
    получать reject
3) Растет:
    queue wait time
    latency
    вероятность cascading failure
4) Поэтому thread pool должен иметь запас для:
    spikes
    slow downstream
    кратковременной деградации зависимостей
ИТОГО: потоков д б с запасом на случай тяжелых запросов или всплесков, чтоб их можно было бы разгрести

Из практики самая главная проблема saturation + queueing
т.е. когда:
CPU saturated
thread pool saturated
DB saturated
GC saturated
начинает расти очередь.
А дальше по законам очередей latency растет НЕ линейно, а резко.

NOTE:
после рестарта сервиса JVM warm-up еще не произошел.
Первый запуск: JVM выполняет байт-код через интерпретатор.
    Параллельно она следит, какие методы вызываются чаще всего.
    Когда метод становится "горячим" (hot spot), включается JIT-компилятор (C1 → C2),
    который переводит его в машинный код (native code). Это требует много CPU и времени.
Второй запуск: Все "горячие" методы уже скомпилированы и лежат в Code Cache.
    JVM просто выполняет готовый нативный код. Интерпретатор почти не используется.

6) GC Pressure:
    например, пики до 0.0025 (0.25% от выделенного CPU уходит на GC)
7) Heap Usage:
    Классическая "пила" - GC работает штатно
    После теста память стабилизировалась на ~X MiB

        уровень	              - что копится
        OS socket backlog	  - TCP connections
        Tomcat accept queue	  - accepted sockets
        Tomcat executor queue - requests
        Client-side queue	  - k6 waiting
5) из-за 4.3:
5.1) создается еще больше тредов (up to max)

6) из-за п.5.1:
